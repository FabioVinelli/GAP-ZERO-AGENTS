## Terminal contract
Every task ends in exactly one of: executed action (landing-checked) | validated handoff | explicit block.
A bare report is a failure mode. NEVER claim an action landed without landing-check evidence.
NEVER simulate an action and present it as executed — label all dry runs "SIMULATED — NOT LANDED".

## Outcome (L0)
- Metric: % validated strategies reaching profitable paper P&L within risk limits
- Baseline → target: 0% → 60% by Q4 2026
- Owner (accountable for errors): Head of Quantitative Trading

## Pipeline (insight → action)
Signal Detected → Evidence Logger → Backtest Engine → Paper Execution → Invalidation Check → Human Approval Gate ◇ → Live Execution

## Policy rules (pre-execution, enforced by src/gapzero/policy.py)
- Live order (any size): ESCALATE
- Position size >2% capital per trade: ESCALATE
- Paper trade or backtest: ALLOW
- Ladder skip detected (backtest or paper missing): BLOCK
- Irreversible actions ALWAYS escalate to the human touchpoint. No exceptions, no self-issued approval tokens.

## Human touchpoint (max ONE validation step)
- Design: Single pre-live-execution approval checkpoint; owner reviews audit trail, backtest P&L, paper fill records, risk parameters before live gate opens
- Owner: Head of Quantitative Trading · Detection: Stop-loss breach or invalidation trigger fires page within one intraday cycle; kill switch auto-arms pending owner response · Rollback: Flatten position via kill switch (live); discard fill record (paper); backtest artifacts retained for audit

## PoC boundaries (pre-committed)
- Scope: One strategy class; paper-trade only in cycle 1; live execution gate locked · Duration: 4 weeks or 50 runs, whichever comes first
- KILL: 3 consecutive max-loss breaches OR Sharpe < 0.5 over 20 trades · SCALE: Success metric met + zero critical evaluator failures + full audit coverage · REVIEW: After 50 runs OR first live-execution request, whichever is earlier
